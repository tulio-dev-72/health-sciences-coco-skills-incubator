### Semantic View Rules

A Semantic View **requires at least one DIMENSIONS or METRICS clause** to be queryable. `TABLES` + `RELATIONSHIPS` alone will fail with: `"No queryable expression is defined in the semantic view."`

Reference: https://docs.snowflake.com/en/sql-reference/sql/create-semantic-view

| Clause | Purpose | Required? |
|--------|---------|----------|
| TABLES | Define logical tables with aliases, PKs, comments | Yes |
| RELATIONSHIPS | Define foreign key joins between tables | Optional |
| FACTS | Private/numeric columns (not directly queryable) | Optional |
| DIMENSIONS | Categorical/date columns (queryable) | **At least one DIM or METRIC required** |
| METRICS | Aggregate expressions (COUNT, SUM, AVG, etc.) | **At least one DIM or METRIC required** |
| COMMENT | Description of the semantic view | Optional |

**Clause order matters**: TABLES → RELATIONSHIPS → FACTS → DIMENSIONS → METRICS → COMMENT

⚠️ **CRITICAL — Semantic View DIMENSIONS/METRICS syntax is REVERSED from normal SQL**:

The `CREATE SEMANTIC VIEW` syntax for DIMENSIONS and METRICS is:
```
TABLE_REF.NEW_DIMENSION_NAME AS EXISTING_COLUMN_NAME
         ^^^^^^^^^^^^^^^^^^      ^^^^^^^^^^^^^^^^^^^^
         (you invent this)       (must exist in the table)
```

This is the **OPPOSITE** of standard SQL aliasing (`SELECT column AS alias`). In standard SQL the alias is on the **right** of `AS`; in Semantic Views the new name is on the **LEFT** of `AS` and the existing column is on the **RIGHT**.

**Examples**:
```sql
DIMENSIONS (
    -- TABLE_REF.NEW_NAME AS EXISTING_COLUMN
    DISCHARGE_SUMMARY_V.DS_MRN AS MRN,                     -- ✅ MRN is a real column
    DISCHARGE_SUMMARY_V.DS_PATIENT AS PATIENT_NAME,         -- ✅ PATIENT_NAME is a real column
    RADIOLOGY_REPORTS_V.RR_IMPRESSION AS IMPRESSION          -- ✅ IMPRESSION is a real column
)
METRICS (
    -- TABLE_REF.NEW_NAME AS AGGREGATE_EXPRESSION
    DISCHARGE_SUMMARY_V.PAT_CT AS COUNT(DISTINCT MRN)       -- ✅ MRN is a real column
)
```

**Common mistake** (will fail with `invalid identifier`):
```sql
-- ❌ WRONG — standard SQL alias syntax does NOT work here
DISCHARGE_SUMMARY_V.MRN AS DS_MRN           -- DS_MRN is not a column → invalid identifier
DISCHARGE_SUMMARY_V.PATIENT_NAME AS DS_PAT   -- DS_PAT is not a column → invalid identifier
```

> **Rule**: The **right side** of `AS` in DIMENSIONS must be an existing column name in the table (or an aggregate expression in METRICS). Always verify column names with `DESCRIBE VIEW <view_name>` before defining dimensions.

**When multiple tables share column names** (e.g., MRN, PATIENT_NAME), use unique dimension names on the LEFT:
```sql
DISCHARGE_SUMMARY_V.DS_MRN AS MRN,          -- unique name DS_MRN, references column MRN
OPERATIVE_NOTES_V.ON_MRN AS MRN,            -- unique name ON_MRN, references column MRN
PATHOLOGY_REPORTS_V.PR_MRN AS MRN            -- unique name PR_MRN, references column MRN
```

**Metrics must be aggregate expressions**:
- WRONG: `alias.los AS DATEDIFF(DAY, col1, col2)` — scalar, not aggregate
- RIGHT: `alias.avg_los AS AVG(DATEDIFF(DAY, col1, col2))` — wrapped in aggregate

```sql
CREATE OR REPLACE SEMANTIC VIEW {db}.{schema}.CLINICAL_DOCS_SEMANTIC_VIEW
    TABLES (
        DISCHARGE_SUMMARY AS {db}.{schema}.DISCHARGE_SUMMARY_V
            PRIMARY KEY (DOCUMENT_RELATIVE_PATH)
            COMMENT = 'Discharge summaries with patient info, dates, and diagnosis.',
        PATIENT_MAPPING AS {db}.{schema}.MRN_PATIENT_MAPPING
            PRIMARY KEY (MRN)
            COMMENT = 'Distinct MRN-to-patient-name mapping derived from extracted documents.'
    )
    RELATIONSHIPS (
        DISCHARGE_SUMMARY (MRN) REFERENCES PATIENT_MAPPING
    )
    DIMENSIONS (
        DISCHARGE_SUMMARY.DOCUMENT_PATH AS DOCUMENT_RELATIVE_PATH
            COMMENT = 'Relative path to the source document file',
        DISCHARGE_SUMMARY.DOC_STAGE AS DOCUMENT_STAGE
            COMMENT = 'Snowflake stage where document is stored',
        DISCHARGE_SUMMARY.MEDICAL_RECORD_NUMBER AS MRN
            WITH SYNONYMS = ('MRN', 'Medical Record Number')
            COMMENT = 'Unique medical record number identifying the patient',
        DISCHARGE_SUMMARY.PATIENT_NAME_DIM AS PATIENT_NAME
            WITH SYNONYMS = ('Patient', 'Name')
            COMMENT = 'Full name of the patient (Last, First Middle)',
        DISCHARGE_SUMMARY.ADMISSION_DATE_DIM AS ADMISSION_DATE
            WITH SYNONYMS = ('Admit Date', 'Date Admitted')
            COMMENT = 'Date the patient was admitted to the hospital',
        DISCHARGE_SUMMARY.DISCHARGE_DATE_DIM AS DISCHARGE_DATE
            WITH SYNONYMS = ('Date Discharged')
            COMMENT = 'Date the patient was discharged from the hospital',
        DISCHARGE_SUMMARY.DIAGNOSIS AS DISCHARGE_DIAGNOSIS
            WITH SYNONYMS = ('Diagnosis', 'Diagnoses', 'Discharge Dx')
            COMMENT = 'Diagnoses at discharge including primary and secondary',
        DISCHARGE_SUMMARY.DISPOSITION AS DISCHARGE_DISPOSITION
            WITH SYNONYMS = ('Disposition', 'Discharge To')
            COMMENT = 'Where the patient was discharged to',
        PATIENT_MAPPING.MRN_KEY AS MRN
            COMMENT = 'Medical record number key',
        PATIENT_MAPPING.PATIENT_NAME_MAP AS PATIENT_NAME
            COMMENT = 'Patient name from mapping table'
    )
    METRICS (
        DISCHARGE_SUMMARY.PATIENT_COUNT AS COUNT(DISTINCT MRN)
            COMMENT = 'Count of distinct patients',
        DISCHARGE_SUMMARY.DOCUMENT_COUNT AS COUNT(DISTINCT DOCUMENT_RELATIVE_PATH)
            COMMENT = 'Count of distinct documents',
        DISCHARGE_SUMMARY.AVG_LENGTH_OF_STAY AS AVG(DATEDIFF(DAY, ADMISSION_DATE, DISCHARGE_DATE))
            COMMENT = 'Average number of days between admission and discharge'
    )
    COMMENT = 'Semantic view for clinical document natural language analytics';
```

## Step 2: Create Cortex Agent

**IMPORTANT**: Before creating the agent, query the configured doc types to inject into the orchestration instruction:

```sql
SELECT LISTAGG(DISTINCT DOC_TYPE, ', ') AS CONFIGURED_TYPES
FROM {db}.{schema}.CLINICAL_DOCS_EXTRACTION_CONFIG
WHERE CONFIG_TYPE = 'EXTRACTION';
```

Use the result to populate `{configured_types}` in the agent spec below.

> **Note**: After adding a new document type and running `GENERATE_DYNAMIC_OBJECTS()`, recreate the agent to update its orchestration instruction with the new type list.

### 🛑 MANDATORY STOP — GATE A2: Agent Configuration

Use `ask_user_question` to ask: "What should I name the agent? Any custom instructions?" Provide the recommended name and schema as defaults. **DO NOT create the agent until user confirms name and any custom instructions.**

**Recommend**: Name `CLINICAL_DOCUMENTS_AGENT` in `{db}.AGENTS` schema.

```sql
CREATE SCHEMA IF NOT EXISTS {db}.AGENTS;

CREATE OR REPLACE AGENT {db}.AGENTS.CLINICAL_DOCUMENTS_AGENT
  COMMENT = 'Clinical Documents AI Agent'
  FROM SPECIFICATION
$$
models:
  orchestration: auto
orchestration:
  budget:
    seconds: 30
    tokens: 16000
instructions:
  response: "You are a helpful clinical documents assistant. Always specify page numbers when available."
  orchestration: "For structured data queries (patient info, dates, diagnoses, extracted fields) use CLINICAL_DOCS_ANALYST. It covers only doc types with configured extraction — currently: {configured_types}. If the user asks about a document type not in the Semantic View, use CLINICAL_DOCS_FULL_CONTENT_SEARCH instead to search page content directly. Explain that structured field extraction is not yet configured for that type and suggest the user run the extraction pipeline to add it. For document content search use CLINICAL_DOCS_FULL_CONTENT_SEARCH."
  system: "You help healthcare professionals retrieve and analyze clinical documents."
  sample_questions:
    - question: "What clinical documents are available for this patient?"
    - question: "Which patients have documents from the last 30 days?"
    - question: "What are the key findings for this patient?"
tools:
  - tool_spec:
      type: cortex_analyst_text_to_sql
      name: CLINICAL_DOCS_ANALYST
      description: "Query structured clinical document metadata — patients, diagnoses, dates, and extracted fields."
  - tool_spec:
      type: cortex_search
      name: CLINICAL_DOCS_FULL_CONTENT_SEARCH
      description: "Search full content of clinical documents."
tool_resources:
  CLINICAL_DOCS_ANALYST:
    semantic_view: "{db}.{schema}.CLINICAL_DOCS_SEMANTIC_VIEW"
    execution_environment:
      type: warehouse
      warehouse: {warehouse}
  CLINICAL_DOCS_FULL_CONTENT_SEARCH:
    name: "{db}.{schema}.CLINICAL_DOCS_SEARCH_SERVICE"
    max_results: 4
    title_column: DOCUMENT_RELATIVE_PATH
    id_column: PRESIGNED_URL
$$;
```
